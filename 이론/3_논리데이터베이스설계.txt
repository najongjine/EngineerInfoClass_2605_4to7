* 외부 스키마
 - DB 전체 구조 중에서 특정 *사용자*나 프로그램이 필요한 부분만 보이게 만든 구조
 - *View*(가상 테이블). 특정 사용자에게 필요한것만 보여줄려고 만든 테이블
* 개념 스키마
 - DB 전체의 논리적 구조
 - 어떤 테이블이 있고
   각 테이블에 어떤 컬럼이 있고
   기본키는 무엇이고
   외래키로 어떻게 연결되어 있고
   어떤 제약조건이 있는지
 - ERD, DDL
* 내부 스키마
 - 내부 스키마는 데이터베이스가 물리적 저장장치에 실제로 저장되는 방법을 정의한 스키마다.
 - 학생 데이터는 디스크의 어느 위치에 저장할지
   인덱스를 만들지
   데이터를 어떤 순서로 저장할지
   파일 구조는 어떻게 할지
   압축할지
   블록 크기는 어떻게 할지
 - 인덱스(데이터를 빨리 찾게 해주는 기능), 파일구조

* RDBMS( Relation DBMS)
 - 데이터를 *테이블* 구조로 관리

* 개체타입(Entity Type), Relation
 - 테이블
* 속성(attribute)
 - 필드, column(열)
* 도메인(Domain)
 - 어떤 컬럼에 들어갈 수 있는 값의 범위
   성별 컬럼에 들어갈 수 있는 값
   → 남, 여
* 튜플(Tuple), Relation Instance, record
 - 행
* 차수(Degree): 전체 속성의 개수
* 기수(Cardinality): 천제 튜플의 개수


p222
* 키 : 행을 유일하게 식별할수 있게 해주는 놈
* 슈퍼키(super key)
 - 행을 구분하기위해 칼럼들을 드럽게 많이 세트로 묶음(최소성이 없음)
* 후보키(Candidate Key)
 - 칼럼 하나만 써도 데이터 중복현상 없음 (최소성이 있다)
 - 모든 튜플에 대해 유일성과 최소성을 모두 만족 시켰다.
* 대체키(Alternate Key)
 - 후보키들 중에서 기본키(PK) 선발에 탈락된 놈들
* 기본키(Primary Key)
 - 후보키들 중에서 데이터 식별 칼럼으로 선정된놈.
 - 유일성, 최소성 모두 만족
 - NULL 값을 가질수 없음
* 왜래키(Foreign Key)
 - 다른 테이블의 기본키를 *참조* 하는 속성
 - 왜래키는 테이블에 중복되 있어어 됨

** 외부 스키마 / 개념 스키마 / 내부 스키마는
데이터베이스의 3단계 스키마 구조를 말한다.

반면에 개념 데이터 모델링 / 논리 데이터 모델링 / 물리 데이터 모델링은
데이터베이스를 설계하는 과정, 즉 DB 설계 단계를 말한다. **

** 개념/논리/물리 데이터 모델링 = DB 설계 단계
1. 개념적 설계 (Conceptual Design, 개념 데이터 모델링)  ER Diagram
 
목표: 
현실 세계를 사람 눈에 보이게 정리하는 단계
[학생] ---- 수강 ---- [과목] ---- 담당 ---- [강사]
주요 활동:
관리해야 할 데이터(개체)와 그 속성(값)을 식별합니다.
개체와 속성 간의 관계를 나타내는 ER 다이어그램(개체-관계 모델)을 작성합니다.
트랜잭션 모델링을 정의하여 데이터베이스와 사용자 간의 데이터 교환 단위를 정의합니다.
 

2. 논리적 설계 (Logical Design, 논리 데이터 모델링)   정규화
 
목표: 
개념 모델을 DB 구조로 바꾸는 단계
주요 활동:
학생(학생번호, 이름, 전화번호)
과목(과목번호, 과목명)
강사(강사번호, 이름)
수강(학생번호, 과목번호, 수강일)

3. 물리적 설계 (Physical Design, 물리 데이터 모델링)    CREATE INDEX
 
목표: 
실제로 DBMS에 만들 수 있게 구체화하는 단계
주요 활동:
CREATE TABLE student (
  student_id INT PRIMARY KEY,
  name VARCHAR(30) NOT NULL,
  phone VARCHAR(20),
  INDEX idx_student_name(name)
);
컬럼 자료형
컬럼 크기
NOT NULL
인덱스
테이블명/컬럼명 실제 이름
저장 구조
성능 고려

* 관계의 종류
 - 다이아몬드 그림으로 표시함
 - 관계를 직접 테이블로 만들지는 않음
 - 1:1 관계: 하나의 객체는 하나의 관계를 갖는다 (사람, dna정보)
 - 1:N 관계 : 하나의 객체는 여러개의 관계를 갖는다 (게시글, 댓글)
 - N:M 관계 : n개의 객체는 m개의 관계를 갖는다 (앨범, 가수)

* 정규화
 - 게시글 댓글 처럼 하나의 데이터가 여러개의 자식 데이터를 가지는 경우
   테이블을 분리하고, 관계를 맺어서 데이터를 저장하는 방법
 - 개념적으론 이렇게 쉬운데, 옜날에는 노하우가 없어서, 수학자가 드럽게 어려운
   공식 5개를 만들었어요
* 1정규화   Atomic
 - 하나의 칼럼은 하나의 값만 가져야 한다.  ㅎㅇ, ㅇㅇ , ㅁㅁ  이렇게 , 쓰지 마라
* 2정규화   
 - 부분 함수적 종속성 제거
 - 완전 함수 종속 만족
* 3정규화
 - 이행 함수 종속 제거
 - A->B B->C A->C
* BCNF(Boyce-Codd)
 - 결정자가 후보키아닌 종속 제거
 - 결정자인데 후보키가 아닌것
* 4정규화
 - 다치 종속 제거
* 5정규화 
 - 조인 종속 제거
