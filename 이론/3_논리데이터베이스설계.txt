  그리고 장소제한 그런것도 있습니다.
  하지만 DB는 인원수 제한없이 몇명이든 같이 사용 되고, 원격으로 사용 되요

p217
* Schema: 데이터베이스의 설계도 / 구조표
  예시 : CREATE TABLE student (
            student_id INT PRIMARY KEY,
            name VARCHAR(50),
            age INT,
            phone VARCHAR(20)
         );
* 외부 스키마
 - DB 전체 구조 중에서 특정 *사용자*나 프로그램이 필요한 부분만 보이게 만든 구조
 - *View*(가상 테이블). 특정 사용자에게 필요한것만 보여줄려고 만든 테이블
* 개념 스키마
 - DB 전체의 논리적 구조
 - 어떤 테이블이 있고
   각 테이블에 어떤 컬럼이 있고
   기본키는 무엇이고
   외래키로 어떻게 연결되어 있고
   어떤 제약조건이 있는지
 - ERD, DDL
* 내부 스키마
 - 내부 스키마는 데이터베이스가 물리적 저장장치에 실제로 저장되는 방법을 정의한 스키마다.
 - 학생 데이터는 디스크의 어느 위치에 저장할지
   인덱스를 만들지
   데이터를 어떤 순서로 저장할지
   파일 구조는 어떻게 할지
   압축할지
   블록 크기는 어떻게 할지
 - 인덱스(데이터를 빨리 찾게 해주는 기능), 파일구조


p220
* RDBMS( Relation DBMS)
 - 데이터를 *테이블* 구조로 관리

* 개체타입(Entity Type), Relation
 - 테이블
* 속성(attribute)
 - 필드, column(열)
* 도메인(Domain)
 - 어떤 컬럼에 들어갈 수 있는 값의 범위
   성별 컬럼에 들어갈 수 있는 값
   → 남, 여
* 튜플(Tuple), Relation Instance, record
 - 행
* 차수(Degree): 전체 속성의 개수
* 기수(Cardinality): 천제 튜플의 개수


p222
* 키 : 행을 유일하게 식별할수 있게 해주는 놈
* 슈퍼키(super key)
 - 행을 구분하기위해 칼럼들을 드럽게 많이 세트로 묶음(최소성이 없음)
* 후보키(Candidate Key)
 - 칼럼 하나만 써도 데이터 중복현상 없음 (최소성이 있다)
 - 모든 튜플에 대해 유일성과 최소성을 모두 만족 시켰다.
* 대체키(Alternate Key)
 - 후보키들 중에서 기본키(PK) 선발에 탈락된 놈들
* 기본키(Primary Key)
 - 후보키들 중에서 데이터 식별 칼럼으로 선정된놈.
 - 유일성, 최소성 모두 만족
 - NULL 값을 가질수 없음
* 왜래키(Foreign Key)
 - 다른 테이블의 기본키를 *참조* 하는 속성
 - 왜래키는 테이블에 중복되 있어어 됨
